# SonarCloud Setup

SonarCloud is the cloud-hosted version of SonarQube. It is free for public repositories and integrates directly with GitHub Actions. Once configured, it automatically analyses every push and pull request.

## One-time setup

### 1. Create a SonarCloud account

1. Go to https://sonarcloud.io and click **Log in with GitHub**.
2. Authorise the SonarCloud GitHub App when prompted.
3. SonarCloud automatically creates a **personal organisation** using your GitHub username as the key (e.g. `mario`). You do not need to create anything manually.

### 2. Import the repository

1. Click **+** (top right) → **Analyze new project**.
2. Select your repository from the list and click **Set Up**.
3. When asked about the analysis method, choose **With GitHub Actions**.

### 3. Obtain your keys

After importing, go to the project page on SonarCloud:

- **Organization key** — shown in the URL: `sonarcloud.io/organizations/YOUR_KEY`
- **Project key** — shown in **Administration → General Settings**, or in the URL parameter `id=`

Both are also displayed during the "Set Up" wizard in SonarCloud.

### 4. Update `sonar-project.properties`

Edit the file at the root of the repository and replace the placeholders:

```properties
sonar.projectKey=YOUR_GITHUB_USERNAME_population-growth-simulator
sonar.organization=YOUR_GITHUB_USERNAME
```

Example for a user whose GitHub username is `mario`:

```properties
sonar.projectKey=mario_population-growth-simulator
sonar.organization=mario
```

### 5. Add the GitHub secret

1. In SonarCloud go to **My Account → Security → Generate Token**. Give it a name and copy the value.
2. In your GitHub repository go to **Settings → Secrets and variables → Actions → New repository secret**.
3. Name: `SONAR_TOKEN`, value: the token you just copied.

### 6. Verify

Push any commit to `main`. The CI workflow (`.github/workflows/ci.yml`) will run and the SonarCloud step will upload results. The analysis appears at `sonarcloud.io/project/overview?id=YOUR_PROJECT_KEY`.

## What SonarCloud checks

- Code smells and maintainability issues
- Bugs and error-prone patterns
- Security hotspots (e.g. hardcoded credentials, injection risks)
- Code coverage (fed from the `coverage.xml` report generated by pytest-cov)
- Duplicated code
